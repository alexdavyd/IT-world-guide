# Примеры артефактов тестирования

## Дефект (баг)
**Определение:** Несоответствие фактического результата выполнения программы ожидаемому результату, выявленное в процессе тестирования.

**Пример:**
*   **ID:** BUG-001
*   **Заголовок:** Невозможно оформить заказ при выборе способа оплаты «Кредитная карта»
*   **Предусловия:** Пользователь авторизован, в корзине есть товары.
*   **Шаги воспроизведения:**
    1. Перейти в корзину.
    2. Нажать «Оформить заказ».
    3. Выбрать способ оплаты «Кредитная карта».
    4. Заполнить данные карты (валидные).
    5. Нажать «Оплатить».
*   **Фактический результат:** Появляется ошибка «Не удалось обработать платеж».
*   **Ожидаемый результат:** Заказ успешно оформлен, деньги списаны.
*   **Окружение:** Chrome 120, Windows 11, production.
*   **Приоритет:** P1 (Высокий)
*   **Серьёзность:** Критическая

## Тест-кейс
**Определение:** Артефакт, описывающий набор входных данных, шагов выполнения и ожидаемых результатов с целью проверки конкретной функции или характеристики системы.

**Пример:**
*   **ID:** TC-LOGIN-001
*   **Заголовок:** Проверка успешного входа с валидными учетными данными
*   **Предусловия:** Пользователь зарегистрирован, браузер открыт на странице входа.
*   **Шаги:**
    1. Ввести валидный email (test@example.com).
    2. Ввести валидный пароль (Qwerty123).
    3. Нажать кнопку «Войти».
*   **Ожидаемый результат:** Пользователь перенаправляется на главную страницу, отображается приветствие с его именем.

## Тестовый сценарий (test scenario)
**Определение:** Высокоуровневое описание того, что нужно протестировать. Обычно представляет собой совокупность связанных тест-кейсов для проверки определённой функциональности или пользовательского пути.

**Пример:**
**Сценарий:** Проверка функциональности «Восстановление пароля»
- Успешная отправка ссылки для сброса на зарегистрированный email.
- Попытка сброса для незарегистрированного email — ошибка.
- Проверка срока действия ссылки для сброса (24 часа).
- Проверка возможности установить новый пароль.
- Проверка входа с новым паролем.

## Набор тестов (test suite)
**Определение:** Совокупность тест-кейсов, объединённых по какому-либо признаку (например, для проверки одного модуля, функциональности или этапа тестирования).

**Пример:**
**Набор тестов:** «Модуль оплаты»
- **TC-PAY-001:** Оплата кредитной картой (успешно)
- **TC-PAY-002:** Оплата кредитной картой (недостаточно средств)
- **TC-PAY-003:** Оплата через PayPal (успешно)
- **TC-PAY-004:** Оплата через PayPal (отмена платежа)
- **TC-PAY-005:** Оплата наличными при получении (только для самовывоза)
- **TC-PAY-006:** Проверка отображения истории платежей в личном кабинете

## Покрытие тестами (test coverage)
**Определение:** Метрика, показывающая, какая часть кода или требований проверена тестами. Используется для оценки полноты тестирования.

**Примеры:**
- **Покрытие требований:** Из 100 функциональных требований автоматизированными тестами покрыто 85 (85%).
- **Покрытие кода:** Инструменты анализа показывают 92% покрытия строк кода (line coverage) для модуля «Корзина».

## Приоритет (priority)
**Определение:** Срочность исправления дефекта с точки зрения бизнеса или потребностей заказчика. Определяет очерёдность выполнения задач.

**Примеры:**
- **P1 (Высокий):** Дефект блокирует работу ключевой функции, необходимо исправить в ближайшее время.
- **P2 (Средний):** Дефект влияет на функцию, но есть обходной путь; исправление можно запланировать в следующий релиз.
- **P3 (Низкий):** Косметический дефект, не влияет на функциональность; исправить, когда будет время.

## Серьёзность (severity)
**Определение:** Степень влияния дефекта на работоспособность системы. Показывает, насколько сильно ошибка мешает пользователю.

**Примеры:**
- **Блокирующий (Blocker):** Система не может функционировать, критическая функция недоступна.
- **Критический (Critical):** Основная функция работает некорректно, но систему можно использовать частично.
- **Значительный (Major):** Функция работает, но с ограничениями, или не соответствует ожиданиям.
- **Незначительный (Minor):** Косметическая ошибка, не влияющая на использование (опечатка, неверный цвет).
- **Тривиальный (Trivial):** Очень мелкая ошибка, почти не заметна.

## Модульное тестирование (unit)
**Определение:** Проверка отдельных компонентов или модулей исходного кода (функций, классов, методов) изолированно от остальной системы.

**Пример:**
Проверка метода `calculateDiscount(amount, userTier)`, который вычисляет скидку в зависимости от суммы покупки и статуса пользователя. Написано несколько тестов:
- Вход: amount = 1000, userTier = "gold" → ожидаемый результат: 200 (скидка 20%).
- Вход: amount = 500, userTier = "silver" → ожидаемый результат: 50 (скидка 10%).
- Вход: amount = 100, userTier = "new" → ожидаемый результат: 0 (нет скидки).

## Интеграционное тестирование
**Определение:** Проверка взаимодействия между несколькими модулями или компонентами системы.

**Пример:**
Проверка корректности передачи данных между модулем «Корзина» и модулем «Оплата» при оформлении заказа. Тест создаёт заказ в корзине, передаёт его в платёжный шлюз и проверяет, что после успешной оплаты статус заказа меняется на «Оплачен».

## Системное тестирование
**Определение:** Проверка системы в целом на соответствие требованиям. Тестируется сквозная функциональность в среде, максимально приближенной к реальной.

**Пример:**
Проверка полного пользовательского пути: регистрация нового пользователя, добавление товара в корзину, оформление заказа с выбором способа доставки и оплаты, получение подтверждения на email, отслеживание статуса заказа в личном кабинете.

## Приёмочное тестирование
**Определение:** Проверка готовности продукта заказчиком или пользователями. Цель — подтвердить, что система соответствует бизнес-требованиям и готова к эксплуатации.

**Пример:**
Заказчик выполняет smoke-тест ключевых сценариев, прописанных в договоре: создание отчёта по продажам за произвольный период, выгрузка отчёта в Excel, отправка его на печать. Тест считается пройденным, если все операции выполняются без ошибок.

## Функциональное тестирование
**Определение:** Проверка функциональных требований — того, что система делает то, для чего она предназначена.

**Пример:**
Тестирование функции поиска товаров по названию:
- Поиск существующего товара «Смартфон» → выдаётся список, содержащий этот товар.
- Поиск по части названия «фон» → выдаются все товары, содержащие «фон».
- Поиск по несуществующему названию «asdfgh» → сообщение «Ничего не найдено».

## Нефункциональное тестирование
**Определение:** Проверка атрибутов качества системы: производительности, безопасности, удобства использования, совместимости и др.

**Примеры:**
- **Нагрузочное:** 1000 одновременных пользователей выполняют поиск товара; проверяется, что время ответа сервера не превышает 2 секунд.
- **Стрессовое:** Резкое увеличение нагрузки до 5000 пользователей; проверяется, что система не падает, а при снижении нагрузки возвращается к нормальной работе.
- **Безопасности:** Попытка внедрения SQL-кода в поля ввода (например, `' OR '1'='1`); проверка, что данные не скомпрометированы.
- **Удобства использования (Usability):** Оценка, насколько легко новый пользователь может найти кнопку «Оформить заказ» без подсказок.
- **Совместимости:** Открытие сайта в разных браузерах (Chrome, Firefox, Safari) и на мобильных устройствах; проверка, что вёрстка не «плывёт».

## Регрессионное тестирование
**Определение:** Проверка, что новые изменения (фичи, исправления) не сломали существующую функциональность.

**Пример:**
После добавления новой формы оплаты «Криптовалюта» прогоняется регрессионный набор тестов, включающий все ранее написанные тест-кейсы для оплаты кредитными картами и PayPal, чтобы убедиться, что они по-прежнему работают.

## Дымовое тестирование (smoke)
**Определение:** Быстрая проверка самой критической функциональности после получения новой сборки, чтобы решить, можно ли принимать сборку в дальнейшее тестирование.

**Пример:**
После установки новой версии приложения на тестовый стенд проверяются:
- Запускается ли приложение (не падает при старте).
- Возможность авторизоваться под тестовым пользователем.
- Открывается ли главная страница.
Если хотя бы одна из этих проверок не проходит, сборка возвращается разработчикам.

## Исследовательское тестирование
**Определение:** Одновременное изучение продукта, проектирование и выполнение тестов без жёстких сценариев, с целью поиска неочевидных дефектов.

**Пример:**
Тестировщик начинает работу с новым модулем «Календарь событий». Он не имеет готовых тест-кейсов, а просто кликает по интерфейсу, пробует создать событие на прошедшую дату, на бесконечную дату, меняет часовые пояса, импортирует события из файла. В процессе он замечает, что при переключении часового пояса время события сдвигается неправильно — создаётся баг-репорт.

## Ручное тестирование
**Определение:** Тестирование, при котором тест-кейсы выполняются человеком вручную без использования средств автоматизации.

**Пример:**
Тестировщик вручную проходит по тест-кейсу TC-LOGIN-001 (ввод логина/пароля, нажатие кнопки), визуально проверяет, что приветствие отобразилось и имя пользователя совпадает с ожидаемым. Все действия выполняются руками, результат фиксируется в отчёте.

## Автоматизированное тестирование
**Определение:** Тестирование, при котором выполнение тест-кейсов и проверка результатов происходят с помощью программных средств, без участия человека.

**Пример:**
Написан автотест на Selenium WebDriver, который запускает браузер, вводит логин и пароль, нажимает кнопку входа, получает текст приветствия и сравнивает его с ожидаемым значением. Тест выполняется автоматически при каждой сборке, результат (успех/неудача) отправляется в систему отчётности.
